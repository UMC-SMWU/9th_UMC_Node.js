## 💥 데이터베이스 정규화 💥

**𓇼 데이터베이스 정규화(Database Normalization)**
: 관계형 데이터베이스에서 **데이터의 중복을 최소화하고 데이터의 일관성과 무결성을 확보하기 위해 테이블을 구조화하는 과정**

**정규화의 주요 목표**

- **데이터 중복 최소화**: 동일한 데이터가 여러 곳에 저장되는 것을 방지하여 저장 공간을 효율적으로 사용함
- **데이터 무결성 향상**: 데이터가 한 곳에서만 관리되므로, 수정 시 모든 관련 데이터가 일관성 있게 변경되어 데이터의 정확성이 높아짐
- **이상 현상(Anomaly) 방지**: 데이터 조작 시 발생할 수 있는 문제를 해결
  - **삽입 이상**: 불필요한 데이터 없이는 새로운 데이터를 추가할 수 없는 문제
  - **갱신 이상**: 중복된 데이터 중 일부만 수정되어 데이터의 일관성이 깨지는 문제
  - **삭제 이상**: 특정 정보를 삭제하면 원치 않는 다른 정보까지 함께 삭제되는 문제

**정규화의 종류**

- **1NF**: 테이블의 모든 컬럼(속성) 값이 원자값(Atomic Value)을 갖도록 하는 것
- **2NF**: 제1정규형을 만족하고, 기본키(Primary Key)의 일부에만 종속되는 컬럼이 없도록 하는 것 (완전 함수 종속)
- **3NF**: 제2정규형을 만족하고, 기본키가 아닌 다른 일반 컬럼에 종속되는 컬럼이 없도록 하는 것 (이행적 함수 종속 제거)

## 💥 인덱스 💥

**𓇼 인덱스(Index)**
: 데이터베이스 테이블의 검색 속도를 높이기 위한 자료구조

**인덱스의 장단점**

- 장점
  - **SELECT 쿼리의 검색 속도 향상**: WHERE 절이나 JOIN과 같은 조건 검색에서 월등한 성능을 보임
  - **시스템 부하 감소**: 전체 데이터를 읽지 않으므로 I/O(입출력)가 줄어들어 시스템의 전반적인 성능이 향상됨.
- 단점
  - **추가 저장 공간 필요**: 인덱스는 원본 데이터와 별도의 공간을 차지하므로, 데이터베이스 크기가 커짐. (보통 테이블 크기의 약 10% 정도)
  - **데이터 변경 작업(INSERT, UPDATE, DELETE) 속도 저하**: 데이터가 변경될 때마다 인덱스 파일도 함께 수정되어야 하므로 이 작업들의 성능이 느려짐
  - **잘못된 인덱스 설계 시 성능 악화**: 데이터의 분포나 쿼리의 종류를 고려하지 않고 무분별하게 인덱스를 생성하면 오히려 성능이 저하될 수 있음

**인덱스를 사용하면 좋은 경우**

- 조회 작업이 빈번하고 데이터 변경작업이 상대적으로 적은 컬럼
- JOIN이나 WHERE 또는 ORDER BY에 자주 사용되는 컬럼

## 💥 ORM VS Raw SQL 💥

ORM과 Raw SQL은 데이터베이스와 상호작용하는 두 가지 다른 접근 방식

**ORM은 프로그래밍 언어의 '객체'를 사용해 데이터베이스를 조작**하게 해주는 도구이고, **Raw SQL은 개발자가 직접 SQL 쿼리문을 작성**해서 데이터베이스에 요청을 보내는 방식

| 구분          | ORM (Object-Relational Mapping)                                     | Raw SQL (순수 SQL)                                              |
| ------------- | ------------------------------------------------------------------- | --------------------------------------------------------------- |
| **개념**      | 객체와 데이터베이스 테이블을 자동으로 매핑(연결)                    | 개발자가 직접 SQL 쿼리문 작성                                   |
| **사용 방식** | 프로그래밍 언어의 메서드나 함수 호출 (e.g., `user.save()`)          | SQL 문법 그대로 사용 (e.g., `INSERT INTO ...`)                  |
| **장점**      | **생산성 향상**, 코드 가독성 좋음, **데이터베이스 종류에 비종속적** | **최상의 성능**, 복잡하고 세밀한 쿼리 작성 가능, 완전한 제어권  |
| **단점**      | 상대적으로 느릴 수 있음, 복잡한 쿼리 작성의 어려움                  | **생산성 저하**, 코드의 직관성 하락, **SQL 인젝션 공격에 취약** |
| **추천 상황** | 빠른 개발이 필요할 때, 간단한 CRUD 작업이 많을 때                   | 최고의 성능이 필요할 때, 복잡한 통계/분석 쿼리가 필요할 때      |

→ 일반적인 CRUD(생성, 읽기, 수정, 삭제) 작업은 ORM을 사용하여 개발 속도를 높이고, 매우 중요하고 복잡한 성능 최적화가 필요한 부분은 직접 쿼리를 작성
