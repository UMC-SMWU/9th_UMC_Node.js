## 💥 외래키 💥

☑️ **외래 키** ⇒ 다른 테이블의 기본키를 참조하는 컬럼으로, 두 테이블 간의 **관계 설정**을 위해 사용.

☑️ 데이터베이스의 **일관성을 유지**하는데 중요한 역할

☑️ 클러스터형 또는 비클러스터형 인덱스를 자동으로 생성할 수 없음

☑️ 한 테이블 내에 여러개의 외래키를 가질 수 있음

☑️ 부모테이블의 기본키(PK), 고유키(UK)를 외래키(FK)로 참조할 수 있음

                 📌 고유키(UK)를 외래키로 참조하는 경우 ( 일반적인 테이블관계 X )
               1.  기본키(PK) 대신 다른 식별 가능한 값이 더 의미가 있는 경우
               2.  데이터가 변경될 가능성이 있는 경우
                 ( 기본키는 보통 변경하면 안 되지만, 고유키는 특정 경우 변경될 수 있음)
               3.  기본적으로 INT형인 기본키(PK)보다 문자열이 더 적합한 경우

## 💥 기본키 💥

☑️ **슈퍼 키(Super Key) : 유일성**을 만족하는 키

☑️ **후보 키 (Candidate Key) : 유일성**과 **최소성**을 만족하는 키 ( ex. 주민등록번호, 학번, 사원번호 )

☑️ **기본 키 (Primary key) :** 후보 키에서 선택 된 키로 NULL 값 허용X , 중복된 값X
→ 테이블 내에서 특정 레코드(행)를 식별하기 위해 사용

## 💥 ER 다이어그램 💥

☑️ **ER다이어그램** ⇒ ( **개념적 데이터 모델** )

데이터베이스 내에서 서로 다른 엔티티가 어떻게 관련되는지 설명하는 데이터 모델을 시각적으로 표현한 것

☑️ **ER다이어그램 용도**

**데이터베이스 설계 및 데이터 모델링:** 비즈니스 분석가와 데이터베이스 엔지니어는 ER 다이어그램을 데이터 모델링 도구로 사용하여 조직에 필요한 데이터베이스의 범위를 평가한 다음 데이터 저장 방법을 계획하며, 데이터 엔지니어가 전체 시스템을 개념화하고 오류 가능성을 줄이는 데 도움이 됨.

**데이터베이스 문제 해결:** 기존 데이터베이스를 ERD와 비교하면 문제를 일으킬 수 있는 데이터베이스 설계 실수를 파악할 수 있음. 수많은 테이블이 포함된 복잡한 데이터베이스에는 디버깅 프로세스를 위해 광범위한 SQL 지식이 필요함. ERD는 데이터베이스를 요약하여 엔지니어가 잠재적인 오류를 신속하게 식별할 수 있게 지원함

## 💥 복합키 💥

☑️ **복합 키** ⇒ ( 두개 이상의 컬럼을 조합하여 PK로 사용하는 것 )

☑️ **단점**

**1ͦ. 인덱스에 좋은 영향을 주지 못함
:** PK로 지정한 컬럼은 자동으로 인덱스가 생성되는데, 복합 키를 통해 조회하면 인덱스 스캔이 제대로 수행되지 않을 수 있음.
인덱스를 구성할 때에도 카디널리티가 높은 것부터 낮은 순으로 구성하는게 좋음.

**2ͦ. FK 관계를 생성 할 때 사이드이펙트가 큼
:** 복합 키에 해당하는 필드 모두 자식테이블에 속해야 함.

**3ͦ. 제약 조건을 변경 시 PK 전체 수정이 필요함
:** 복합키에 필요한 필드들의 제약조건이 변경된다면 PK전체를 수정해야 할 수도 있으며, 다른 테이블과 FK관계를 맺고 있는 경우, 해당 테이블의 내용도 수정해야 함.

---

☑️  **복합 키를 사용하는 경우**

- **특정 데이터를 식별하는 게 의미가 없는 도메인의 경우**
  : 단순히 수치와 같은 조회를 위한 통계성 데이터들에 가까운 경우

- **현실 세계의 복잡한 비즈니스 규칙을 데이터베이스에 표현하고자 하는 경우**
  : 비즈니스적 규칙을 데이터베이스에 적용하는 경우에는 자연 키(Natural Key)의 의미를 더 살릴 수 있고, 그로 인해 데이터의 자체 문서화 효과를 얻을 수 있음. 또한 실수로 중복 데이터가 들어가는 것을 원천
  차단하여, 데이터 정합성 보장을 더 확실하게 지켜낼 수 있음.

## 💥 연관관계 💥

☑️ 연관 관계 ⇒ ( 두 도메인인 객체와 테이블이 서로 논리적인 의미를 갖고 양쪽을 참조하는 관계 )

☑️ 연관 관계 매핑 ⇒ 객체의 참조와 테이블의 외래 키를 매핑하는 것

☑️ 연관 관계 주인
: 두 객체가 양방향 관계 (A → B, B → A)를 맺을 때, 연관 관계를 주인을 지정하여 Owner만 두 객체 사이에서 추가, 수정, 삭제를 가능하게 함. 연관관계의 주인이 아니면 조회만 가능하여 불필요한 수정을 막음.
외래 참조를 가지는 쪽은 주인으로 지정하며, 연관 관계의 주인이 아닌 객체는 @mappedBy 속성으로 주인을 지정

☑️ **다대일 단방향** **(N:1)**
가장 흔하게 사용되는 연관관계. 연관관계의 주인은 (N)이며,
(N)인 쪽에 `@ManyToOne` 만 추가

```java
@Entity
public class Post {
    @Id @GeneratedValue
    @Column(name = "POST_ID")
    private Long id;

    @Column(name = "TITLE")
    private String title;

    @ManyToOne
    @JoinColumn(name = "BOARD_ID")
    private Board board;
}

@Entity
public class Board {
    @Id @GeneratedValue
    private Long id;
    private String title;
}
```

☑️ **다대일(N:1) 양방향**
(1) 쪽에 `@OneToMany` 를 추가하고 연관 관계의 주인을 `mappedBy` 로 지정

```java
@Entity
public class Post {
    @Id @GeneratedValue
    @Column(name = "POST_ID")
    private Long id;

    @Column(name = "TITLE")
    private String title;

    @ManyToOne
    @JoinColumn(name = "BOARD_ID")
    private Board board;
}

@Entity
public class Board {
    @Id @GeneratedValue
    private Long id;
    private String title;

    @OneToMany(mappedBy = "board")
    List<Post> posts = new ArrayList<>();
}
```

☑️ **일대일(1:1) 양방향**
`@OneToOne` 추가하고 `mappedBy` 설정

```java
@Entity
public class Attach {
    @Id @GeneratedValue
    @Column(name = "ATTACH_ID")
    private Long id;
    private String name;

    @OneToOne(mappedBy = "attach")
    private Post post;
}
```

**📌 일대다(1:N) 단방향, 양방향 연관 관계 매핑이 필요한 경우 다대일(N:1) 양방향 연관관계로 매핑하자**
다대다(N:M) 경우 연결테이블을 추가하여 일대다, 다대일 관계로 풀어서 사용

## 💥 정규화 💥

☑️ 중복된 데이터를 허용하지 않음으로써 무결성(Integrity)를 유지하며 이상현상을 없애는 과정
→ 데이터의 일관성, 최소한의 데이터 중복, 최대한의 데이터 유연성을 위한 방법

☑️ **장점**

- 이상현상 제거
- 정규화된 데이터베이스 구조에서는 새로운 데이터 형의 추가로 인한 확장 시,
  그 구조를 변경하지 않아도 되거나 일부만 변경
- 데이터베이스와 연동된 응용 프로그램에 최소한의 영향만을 미치게 되어 응용프로그램의 생명을 연장시킴
- 저장 공간의 최소화 (용량 감소), 데이터 구조의 안정성 및 무결성 유지

☑️ **단점**

- 릴레이션의 분해로 인해 릴레이션 간의 JOIN연산이 많아짐 → 성능저하

☑️ 정규화 종류

1NF
• 테이블의 칼럼이 원자값(Atomic value, 하나의 값)을 갖도록 테이블을 분해하는 것

2NF
• 제1정규화를 진행한 테이블에 대해 완전 함수 종속을 만족하도록 테이블을 분해하는 것

3NF
• 제2정규화를 진행한 테이블에 대해 이행적 종속을 없애도록 테이블을 분해하는 것

## 💥 반정규화 💥

☑️ 성능을 위해 데이터 중복을 허용하는 것

☑️ 장점

- 데이터를 빠르게 조회할 수 있음
- 조인을 제거하기 때문에 검색 시간 최적화

☑️ 단점

- 데이터의 삽입, 삭제, 수정 등 갱신 시 비용이 높아짐
- 데이터간의 일관성이 깨질 수 있음 (서로 다른 데이터가 저장될 수 있음)
- 많은 저장 공간이 필요함

☑️ **반정규화가 필요한 경우**

**1ͦ.** 자주 사용되는 테이블에 접근하는 프로세스 수가 많고, 항상 일정범위만 조회하는 경우

**2ͦ.** 통계성 프로세스에 의해 통계 정보를 별도로 필요로 하는 경우

**3ͦ.** 테이블이 지나치게 많은 조인이 걸려 조회가 어려운 경우
